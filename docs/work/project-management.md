# 项目推进模式

来源：[我所经历过的美国科技公司中几种不同项目推进形式和企业文化](https://blog.douchi.space/tech-project-management/?utm_source=rss)

项目推进的关键，不是谁最忙，而是谁决定 `做什么`、谁决定 `怎么做`、谁负责 `unblock`。

## 常见角色

| 角色 | 主要职责 |
| --- | --- |
| `PM` | 决定做什么、为什么做 |
| `UXD` | 决定交互和界面 |
| `EM` | 管人、排资源、做组织协调 |
| `TL` | 定技术方向、做技术裁定 |
| `TPM` | 协调跨职能进度、风险和依赖 |
| `Eng lead` | 具体执行负责人，推动落地 |

## 四个稳定模式

### 1. TPM 驱动

- 适合大型跨团队项目
- 重点是进度、依赖和交接
- 项目负责人更像 `TPM`

### 2. PM / 数据驱动

- 适合产品实验较强的团队
- `PM` 定目标，`DA/DS` 定指标，`SWE` 实现
- 项目负责人更像 `PM + 数据分析`

### 3. TL 驱动

- 适合小团队、单一技术栈、scope 较清晰的项目
- `TL` 直接定方向，工程师按技术方案推进
- 项目负责人更像 `TL`

### 4. Eng lead 驱动

- 适合多平台、多依赖、组织松弛的项目
- `PM/EM` 定大方向，具体执行由 `Eng lead` 推
- 项目负责人更像 `Eng lead`

## 结论

- 角色分工越清晰，沟通成本越低
- 项目越大，越需要显式的依赖管理
- 工程师体验往往取决于 `谁在推动`，而不只是 `做什么`
